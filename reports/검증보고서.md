# 검증 보고서 — 실험 신뢰성 검증 전체 정리

## 개요

각 실험 파이프라인에서 수행한 검증 실험들을 통합 정리한다.
검증 목적: Data Leakage 방지, 과도한 성능의 원인 규명, 임계값 설정의 타당성 확인.

---

## 1. RUL — Data Leakage 검증

### 검증 내용
`operating_hours` 피처가 정답의 역함수인지 실험으로 증명.

### 수학적 관계
```
operating_hours = file_idx / max(file_idx)
rul_ratio       = 1 - file_idx / max(file_idx)
∴ operating_hours = 1 - rul_ratio  (모든 데이터셋에서 성립)
```

### 결과

| 구분 | Accuracy | F1-Macro |
|------|----------|----------|
| operating_hours 포함 (기준선) | ~99.99% | ~99.99% |
| operating_hours 제거 (단계 1) | ~40% | 29.43% |
| 낙폭 | **-60%p** | **-70%p** |

### 판정
**Leakage 확정.** 단일 피처 제거로 성능이 60%p 이상 하락. 해당 피처가 사실상 정답지 역할.

### 부수 발견
Leakage가 환경보정 효과도 가리고 있었음:
- Leakage 있을 때 환경보정 효과: -0.05%p (없는 것으로 보임)
- Leakage 제거 후 환경보정 효과: +1.75%p (진짜 효과 드러남)
- HI 라벨 + balanced weight 이후: +5.95%p (최대 효과)

**백업**: `D:\project\예지보전_v2\증명_RUL_operating_hours제거\`

## 2. 이상탐지 — HI 소스 피처 Leakage 검증

### 검증 내용
HI 기반 라벨링 도입 후, HI를 계산하는 소스 피처(vibration_rms, short_trend, rolling_trend)가 피처에 포함되어 모델이 라벨을 역산할 수 있는지 검증.

### HI 계산 구조
```
HI = f(vibration_rms)
라벨 = (HI > 0.7 → 정상, HI ≤ 0.6 → 이상)
∴ vibration_rms → HI → 라벨  (순환 가능성 존재)
```

### 실험 설계
16피처 → 13피처 (vibration_rms, short_trend, rolling_trend 제거)

### 결과

| 모델 | 16피처 (HI 소스 포함) | 13피처 (HI 소스 제거) | 차이 |
|------|---------------------|---------------------|------|
| IsolationForest | AUC 0.988 / Det 97.2% | AUC 0.984 / Det 96.4% | AUC -0.004 / Det -0.8%p |
| Autoencoder | AUC 0.985 / Det 93.9% | AUC 0.984 / Det 92.7% | AUC -0.001 / Det -1.2%p |
| OneClassSVM | AUC 0.986 / Det 96.0% | AUC 0.983 / Det 94.8% | AUC -0.003 / Det -1.2%p |

### 판정
**Leakage 아님 확정.** HI 소스 피처 제거 후 AUC 하락 0.001~0.004에 불과. 나머지 13개 피처(kurtosis, spectral_energy 등)가 독립적으로 정상/이상을 구분하는 충분한 정보를 보유.

**최종 결정**: 16피처 복원 (RUL과 동일한 피처 구성으로 시스템 일관성 확보)

**백업**: `D:\project\예지보전_v2\증명_AD_HI분리_leakage검증\`

## 3. 이상탐지 — 난이도 검증 (경계 좁힘)

### 검증 내용
HI 기반 분리에서 AUC 0.988이 정상/이상 경계가 넓어(0.3) 쉬운 과제였던 것인지, 실제 모델 능력인지 검증.

### 실험 설계
이상 임계값을 ≤0.4(간격 0.3) → ≤0.6(간격 0.1)으로 변경하여 난이도 상향.

```
변경 전: 정상 HI > 0.7 / 이상 HI ≤ 0.4  (간격: 0.3)
변경 후: 정상 HI > 0.7 / 이상 HI ≤ 0.6  (간격: 0.1)
→ 초기 열화(HI 0.4~0.6) 구간도 이상으로 포함, 경계 더 모호
```

### 결과

| 모델 | ≤0.4 (간격 0.3) | ≤0.6 (간격 0.1) | AUC 변화 |
|------|----------------|----------------|---------|
| IsolationForest | AUC 0.988 / Det 97.2% | AUC 0.970 / Det 80.2% | -0.018 |
| Autoencoder | AUC 0.985 / Det 93.9% | AUC 0.977 / Det 87.9% | -0.008 |
| OneClassSVM | AUC 0.986 / Det 96.0% | AUC 0.941 / Det 81.2% | -0.045 |

### 판정
**모델 실력 확인.** 정상/이상 간격을 1/3로 줄여도 AUC 0.94~0.97 유지. 초기 열화(MEDIUM 단계)도 80~88% 탐지 가능.

**추가 결정**: ≤0.6 기준을 최종 운영 임계값으로 채택
- 스토리텔링 시나리오에서 MEDIUM 단계 조기 감지 필수
- ≤0.4 기준이면 HIGH 단계 이후 감지 → 예지보전 목적 달성 불가
- T분 지속 조건으로 오탐 방지

**백업**: `D:\project\예지보전_v2\증명_AD_경계좁힘검증\`

## 4. RUL — 사분위 라벨 기준 실패 검증

### 검증 내용
HI 하드코딩 기준(0.7/0.4/0.15) 대신 데이터 기반 사분위(q75/q50/q25)로 클래스를 균등 배분하는 방안 검증.

### 실험 설계
사분위 기준 적용 시 실제 HI 값: q75=0.998, q50=0.990, q25=0.956

### 결과

| 구분 | 최고 F1 | Overfit Gap |
|------|--------|------------|
| 하드코딩 기준 (단계 7) | 50.79% | 3.3% |
| 사분위 기준 (단계 8) | 42.50% | 37.7% |
| 변화 | **-8.29%p** | **+34.4%p** |

### 판정
**실패. 7단계로 복원 완료.**

실패 원인:
- 사분위 기준 HI 범위(0.956~0.998)가 극히 좁음
- 이 범위의 차이는 정상 베어링 내 측정 노이즈 수준 (물리적 열화 차이 아님)
- 모델이 노이즈를 분류 신호로 학습 → Gap 30%+ 급증 (심각한 과적합)
- 교훈: 클래스 균형을 위한 경계 조정은 물리적 의미를 반드시 확인해야 함

**백업**: `D:\project\예지보전_v2\증명_RUL_사분위실패_되돌림\`

## 5. RUL — XJTU-SY 도메인 이질성 검증

### 검증 내용
XJTU-SY 15개 베어링을 RTF용으로 추가(22→37 bearings)하여 데이터 확장 효과 검증.

### 실험 설계
- `load_xjtu_rtf()` 추가: H+V 축 결합(combined = sqrt(H²+V²)), fs=25.6kHz
- `load_all_rtf()`에 등록하여 LOBO-CV에 포함

### 결과

| 모델 | 22 bearings F1 | 37 bearings F1 | 변화 |
|------|---------------|---------------|------|
| XGBoost | 49.42% | 46.29% | **-3.1%p** |
| LightGBM | 46.43% | 45.67% | -0.8%p |
| RandomForest | 48.80% | 42.09% | **-6.7%p** |

### 판정
**실패. load_all_rtf에서 XJTU-SY 제거 완료.**

실패 원인:
- XJTU-SY 운전 조건: 2100~2400rpm, 10~12kN (기존 4개 데이터셋과 이질적)
- LOBO-CV에서 XJTU-SY가 test fold에 배정되면 train 분포와 크게 달라 예측 불가
- 단순 데이터 확장 무효 — 도메인 적응(Domain Adaptation) 기법 필요
- `load_xjtu_rtf()` 함수는 코드에 잔존 (향후 활용 가능)

**백업**: `D:\project\예지보전_v2\증명_RUL_XJTU_RTF추가\`
